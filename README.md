# Словник — ქართული

Локальний Flask-сервер для запису грузинських слів з телефону, поки вчу мову на LingWing.

## Запуск

```
pip install -r requirements.txt
python app.py
```

Або подвійний клік на `start.bat`.

Консоль покаже дві адреси (`https://`, якщо на комп'ютері встановлено mkcert — див. нижче, інакше `http://`):
- `127.0.0.1:5000` — з цього комп'ютера
- `<local-ip>:5000` — з телефону (той самий Wi-Fi). IP може змінюватись після перезавантаження роутера — дивись актуальний в консолі при кожному старті; сертифікат для нової IP перегенерується автоматично.

З телефону відкрий адресу в браузері і додай на головний екран ("Add to Home Screen") — буде як окрема аплікація.

При першому запуску Windows Firewall може спитати дозвіл для Python — треба дозволити для приватних мереж, інакше телефон не з'єднається.

## Тести

```
python tests/run_all.py
```

Сервер має бути запущений (`python app.py`) — тести звертаються до нього по HTTP. Прогонить усі серверні тести (`tests/test_*.py`) і клієнтські (`tests/js/test_*.js`, потрібен Node.js — перевіряють JS-логіку прямо зі `static/js/*.js` через ізольований `eval`, файли конкатенуються в тому самому порядку, що й `<script src>` в `templates/index.html`).

- Усі тести створюють лише власні тестові слова (з префіксом на кшталт `test-api-`, `test-srs-`, `test-imp-`) і прибирають їх за собою в кінці — безпечно ганяти проти реального словника, справжні слова не зачіпаються.
- `tests/test_chat_api.py` перевіряє лише безкоштовні шляхи (конфігурація, валідація) і **ніколи не надсилає справжніх повідомлень до Claude** — щоб прогін тестів не витрачав платні токени.
- `run_all.py` повертає ненульовий код виходу, якщо щось провалилось — зручно перед комітом чи в CI.

## Бекап бази даних

```
python backup_db.py
```

Або подвійний клік на `backup_db.bat`. Створює копію `dictionary.db` у `backups/dictionary-<дата>_<час>.db` через SQLite Online Backup API (`Connection.backup()`) — безпечно запускати, навіть коли сервер уже працює й саме в цей момент щось пишеться в базу (на відміну від простого копіювання файлу). Тека `backups/` не в git.

**Автобекап** — той самий механізм працює й автоматично: фоновий потік у `app.py` (`_auto_backup_loop`) робить бекап раз на `AUTO_BACKUP_INTERVAL` (доба). Свідомо НЕ окрема крон-задача рівня ОС і НЕ бекап на кожну дію в застосунку: перше вимагало б окремого налаштування поза кодом репозиторію (не переноситься автоматично, якщо хтось інший захоче собі цю аплікуху), друге — довелось би чіплятись до кожного write-ендпоінта і сповільнило б кожен дрібний запит. Прив'язаний до РЕАЛЬНОГО часу останнього бекапу (`last_backup_at()` читає мітку часу з імені найновішого файлу в `backups/`), а не до "сервер працює 24 год поспіль" — комп'ютер не завжди увімкнений (dual-boot, вимкнення на ніч): якщо бекапів ще не було, або минуло більше інтервалу (комп'ютер довго був вимкнений) — бекап робиться одразу при старті; якщо минуло менше (частий перезапуск сервера) — фоновий потік чекає залишок часу, щоб не плодити майже-дублікати (`_seconds_until_next_backup()`). Якщо сам бекап падає з помилкою — повторна спроба через хвилину (не через цілу добу), щоб постійна помилка (напр. диск заповнений) не перетворилась на мовчазну багатоденну відсутність бекапів. `prune_backups()` лишає останні `DEFAULT_KEEP` (30) звичайних бекапів, старіші видаляє — але НЕ чіпає `dictionary-before-restore-*.db` (їх свідомо створює `restore_db.py` як разову страховку, ротація не повинна їх мовчки прибирати).

### Відновлення з бекапу

```
python restore_db.py                                    # найновіший бекап
python restore_db.py dictionary-2026-07-09_23-41-05.db   # конкретний файл
```

Або подвійний клік на `restore_db.bat` (без аргументу — відновлює найновіший). **Спершу зупини сервер.** Перед заміною поточна `dictionary.db` не видаляється, а переміщується в `backups/dictionary-before-restore-<дата>_<час>.db` — можна відкотити відновлення, якщо помилився. Питає підтвердження перед дією; коротко перевіряє, що файл справді схожий на базу словника (а не сторонній/пошкоджений файл), перш ніж щось міняти.

## Пошук дублікатів

```
python find_duplicates.py
```

Або подвійний клік на `find_duplicates.bat`. Друкує окрему таблицю на кожну групу слів з однаковим грузинським написанням (id, переклад, приклад, теги, SRS-прогрес, дата) — найбільші групи першими. Групування точно за `georgian` — та сама умова (`WHERE georgian = ?`), за якою чат-інструменти (`edit_word`/`retag_word`) відмовляються редагувати неоднозначне слово, а CSV-імпорт шукає слово без `uuid`; тобто звіт показує рівно ті групи, які насправді заважають застосунку.

Скрипт **нічого не змінює** — база відкривається read-only, тож безпечно запускати навіть при живому сервері. Дублікат не завжди помилка: у грузинській є омоніми (однакове написання, різне значення), тому застосунок при додаванні лише попереджає, але не блокує — рішення (лишити як омоніми, злити чи прибрати зайве) за тобою. Колонка SRS (`рівень ka→uk/uk→ka`, «—» — ще не тренувалось) саме для цього: видно, яку копію ти вже тренував, щоб не видалити її замість порожньої.

## Що вміє

- Додавання слова: грузинська, переклад, приклад/нотатка (необов'язково). Якщо таке саме грузинське написання вже є в словнику — попереджає (з перекладом наявного слова) і питає підтвердження, але не забороняє: у грузинській бувають омоніми (однакове написання, різне значення)
- **Переклад і приклад приховані в списку** — видно лише грузинське слово; тап по ньому показує переклад і приклад (ще один тап — ховає назад). Спонукає спершу пригадати переклад самому, а не просто читати список. Стан розкриття не зберігається в localStorage — завжди приховано заново при відкритті сторінки, але лишається розкритим у межах сесії (не скидається від пошуку, синку тощо)
- Пошук по грузинській і перекладу (миттєвий, працює й офлайн; шукає і в прихованому перекладі теж)
- Редагування та видалення
- Експорт у CSV (кнопка "⬇ CSV") — щоб потім віддати AI на перевірку правильності перекладів/прикладів. Генерується прямо в браузері з локальних даних (Blob + `download`), працює офлайн і не "виносить" зі standalone-PWA на iOS, як зробило б звичайне посилання на серверний маршрут. Містить колонку `uuid` — вона потрібна для імпорту, проси AI її не чіпати
- **Імпорт CSV назад** (кнопка "⬆ CSV") — замикає цикл AI-перевірки: експортував → AI виправив → імпортував. Рядок з відомим `uuid` оновлює наявне слово (SRS-прогрес зберігається), з порожнім/невідомим — створює нове. Захисна сітка від поламаних AI uuid: якщо uuid не знайдено, але в базі рівно одне слово з таким самим грузинським написанням — оновлюється воно, а не створюється дублікат (при кількох омонімах сітка не вгадує і створює нове). Перед застосуванням показується підсумок, повторний імпорт того ж файлу ідемпотентний. Колонки розпізнаються по заголовку, порядок неважливий; обов'язкові лише `georgian` і `translation`. Працює тільки онлайн
- **Теги/категорії** — необов'язкове поле "теги через кому" при додаванні й редагуванні ("їжа", "дієслова", "урок 12"...). Теги нормалізуються (нижній регістр, без дублів, без зарезервованих слів "усі"/"проблемні" — інакше в панелі було б два чипи з однаковим текстом; фільтрація в `normalizeTags()`/`normalize_tags()`, спрацьовує для будь-якого шляху додавання тега — форма, редагування, перейменування, CSV-імпорт, чат-інструмент `add_word`). Над списком з'являється панель чипів з лічильниками: тап фільтрує список, повторний тап знімає фільтр. Теги під самим словом у списку теж клікабельні — так само фільтрують, не треба гортати панель чипів, щоб знайти потрібний (клік зупиняє спливання події, інакше зачепив би й тап-показ перекладу картки; вибраний тег підсвічується на кожному слові так само, як активний чип). Панель тегів прокручується до щойно вибраного чипа (чи назад на початок при знятті фільтра) лише як реакція на саму зміну тега — випадковий пошук чи фоновий синк, поки тег не змінювався, ручний скрол користувача не чіпають. Активний тег обмежує і **повторення** — можна тренувати одну тему окремо. Теги входять у CSV-експорт/імпорт, тож AI може категоризувати словник за тебе
- **Масове перейменування й видалення тега** — коли обраний конкретний тег (фільтр активний), поруч із чипами з'являються кнопки "✎" (перейменувати в усіх словах одразу — зручно виправити одруківку) і "🗑" (прибрати тег з усіх слів без заміни на інший, самі слова лишаються). Обидві операції порівнюють точно по токену, не по підрядку (напр. "їжа" не зачепить "їжа2"); перейменування в наявний тег не створює дублікат. Кнопки "прилипають" (`position: sticky` + `margin-left: auto`, тому завжди притиснуті праворуч, навіть коли тегів мало й скролу немає) до правого краю панелі — не треба гортати до самого кінця, коли тегів уже багато. Тінь зліва від кнопок з'являється лише тоді, коли в поточній позиції скролу справді щось ховається праворуч (`updateTagbarShadow()`: враховує і сам факт переповнення, і те, чи домотано вже до кінця) — оновлюється і після перебудови панелі, і по самій події `scroll`. Потребують з'єднання з сервером
- **Слово дня** — картка над формою додавання: зважено-випадкове слово зі словника, змінюється раз на добу. Слова з нижчим рівнем SRS (гірше вивчені; відсутнє повторення теж рахується як рівень 0 — найвищий пріоритет) отримують вищу вагу і тому трапляються значно частіше, але добре вивчені слова повністю не виключаються — рідко, але теж можуть випасти. Реалізовано алгоритмом Ефраїмідіса — Спіракіса (`wodKey` в `static/js/stats.js`): кожному слову — ключ `u^(1/вага)` з псевдовипадкового `u` (хеш дата+uuid), перемагає максимальний ключ; це дає coректний зважений розподіл БЕЗ повторної випадковості на кожен рендер (детерміновано в межах дня) і, на відміну від попереднього варіанту `sorted[h % words.length]`, вибір не залежить від довжини масиву — додавання нового слова більше не "перекидає" день на випадкове інше слово. Переклад прихований до тапу (маленька вправа на пригадування замість пасивного показу); після тапу лишається відкритим до кінця дня (`localStorage`)
- **Режим друкування відповіді в повтореннях** (типово увімкнений; кнопка "⌨️/👁" в сесії перемикає на класичний показ, вибір запам'ятовується). Друкуєш відповідь → "Перевірити" → вердикт ✅/❌ + правильна відповідь, фінальну оцінку однаково даєш сам (раптом синонім). Перевірка поблажлива: ігнорує регістр/розділові/дужки, зараховує будь-який варіант через кому ("дякую" ≈ "дякую, спасибі"), прощає одну одруківку словам від 5 літер. Порожній ввід + "Перевірити" = просто показати відповідь без вердикту
- **Строгий режим** (кнопка "🔒/🔓" поруч із ⌨️/👁, видима лише в режимі друкування; вибір запам'ятовується) — вимикає прощення одруківок, щоб точно вивчити написання буква в букву (регістр/розділові/дужки й варіанти через кому досі ігноруються — це про орфографію, не про формат)
- **Клавіатура на мобільних не ламає повноекранні вікна** (Повторення/Чат/Нотатки) — фокус на полі вводу блокує скрол фонового списку слів (інакше iOS/Android прокручують сторінку за вікном, а не саме вікно), а `VisualViewport` API підганяє вікно під фактично видиму область над клавіатурою, замість того щоб вікно "виїжджало" вгору за межі екрана
- **Клавіатурне керування повтореннями** (для десктопа): Enter — перевірити/показати відповідь; на екрані оцінки `1` — "не знав", `2` — "знав", Enter — підтвердити вердикт перевірки (якщо друкував відповідь); Escape — закрити сесію. На фінальному екрані: якщо є провалені картки цієї сесії (видима кнопка "🔁 Ще раз провалені") — Enter одразу починає цей міні-раунд; якщо кнопки нема — Enter закриває вікно, як і раніше. Escape на фінальному екрані завжди закриває ("пропустити"), незалежно від наявності кнопки. Підказка клавіш показується внизу картки лише на пристроях з мишею/клавіатурою (`hover: hover`)
- **Коли повернутись за повторенням** — якщо зараз немає ні прострочених, ні нових карток (кнопка "🎓 Повторення" неактивна), під кнопками з'являється рядок "Наступне повторення сьогодні/завтра о ГГ:ХХ" (`nextDueAt()` в `static/js/srs.js` шукає найранішу МАЙБУТНЮ картку серед `reviews`, з урахуванням активного тег-фільтра) — щоб не треба було раз у раз відкривати застосунок "про всяк випадок", просто перевірити, коли справді є сенс повернутись. `due_at` зберігається в UTC, тому перед показом переводиться в локальний час користувача (`formatNextDue()` в `static/js/render.js`)
- **📅 Щоденне нагадування в календар телефону** — кнопка в шапці (`downloadReminder()` в `static/js/main.js`) питає бажаний час і генерує `.ics`-файл (щоденна подія з `RRULE:FREQ=DAILY` і сигналом у момент початку) — телефон відкриває його своїм календарем (Google/Apple Calendar), і далі нагадування живе повністю там, незалежно від того, чи запущений сервер цього застосунку. Свідомо НЕ push-сповіщення: той варіант вимагав би, щоб `app.py` на комп'ютері був увімкнений саме в момент нагадування, що для локального (не хмарного) сервера ненадійно, якщо комп'ютер вимкнений, коли тебе немає вдома
- **💬 Чат з репетитором (Claude)** — вбудований AI-репетитор, який бачить твій словник і SRS-прогрес: розмовляє грузинською з підлаштуванням під рівень, виправляє помилки з поясненням правил, будує вправи на твоїх словах. Історія розмови зберігається в базі, відповіді стрімляться. Працює лише онлайн і потребує API-ключ (див. розділ "Чат: налаштування"). Недописане повідомлення в полі вводу зберігається як чернетка в `localStorage` (`chatDraft`) — переживає закриття вкладки/застосунку, не лише перехід між екранами; прибирається одразу після успішного надсилання. Поле вводу — `textarea`: на десктопі Enter надсилає повідомлення, Shift+Enter додає новий рядок (напр. для списку пунктів); на сенсорних пристроях (де немає Shift+Enter — визначається через `matchMedia("(pointer: coarse)")`) Enter лишається звичайним переносом рядка, а надсилання лише кнопкою ➤, інакше багаторядкове повідомлення набрати було б неможливо. Поле саме розтягується під текст (до ~5 рядків, `CHAT_INPUT_MAX_HEIGHT`), далі — внутрішній скрол, і повертається до одного рядка після надсилання
- **Додавання слів прямо з чату** — якщо попросиш "додай це слово" або сам погодишся зберегти нове слово з розмови, репетитор додає його в словник (tool use, `add_word`: переклад, за потреби приклад і теги). Клієнт після кожної відповіді чату підтягує свіжий стан словника, тож нове слово одразу з'являється в списку й повтореннях без перезавантаження сторінки. Опис поля `example` в інструменті просить репетитора (де це природно) вживати слово в реченні точно в словниковій формі — щоб нові слова одразу підтримували cloze-підказку на повтореннях, без ручного редагування CSV пізніше
- **Форми нерегулярних дієслів через чат** — грузинська дуже нерегулярна (дієслова руху міняють корінь між часовими серіями, форми різняться за особою/числом об'єкта), тож окремої таблиці "неправильних дієслів" у застосунку немає — замість цього репетитор сам пропонує додати ключові форми окремими словами через `add_word` з однаковим тегом `дієслово:<словникова форма>` (напр. усі форми "йти/прийти" — тег `дієслово:წასვლა`), тож вони групуються в один чип у панелі тегів і тренуються разом, без ручного підтримування конвенції тегів самим користувачем. Той самий тег можна додати і до вже наявних слів — інструмент `retag_word` знаходить існуюче слово за точним грузинським написанням і домішує нові теги до наявних (нічого не перезаписує); якщо знайдено кілька слів з однаковим написанням — нічого не змінюється, репетитор просить уточнити
- **Виправлення помилок у вже доданих словах через чат** — інструмент `edit_word` (той самий пошук за точним грузинським написанням, та сама відмова при неоднозначному збігу, що й у `retag_word`) виправляє переклад, приклад і/або теги наявного слова. На відміну від `retag_word`, тут теги **повністю замінюють** наявні (а не домішуються) — бо мета не впорядкувати форми дієслова, а виправити конкретну помилку (напр. забутий переклад чи невірний приклад); передається лише те поле, яке справді треба змінити, решта лишається як є. Репетитору явно сказано не виправляти помилки повторним `add_word` — це створило б дублікат
- **🔔 Останні дії інструментів** — кнопка поруч з 🗑 у шапці чату показує список останніх викликів `add_word`/`edit_word`/`retag_word`/`save_grammar_note`/`get_grammar_note`/`edit_grammar_note` (таблиця `tool_calls`, кожен виклик логується одразу після виконання в `_run_chat_generation`). Заголовок картки короткий ("✓ add_word — ფანჯარა"), тап розгортає деталі: для `edit_word`/`retag_word`/`get_grammar_note`/`edit_grammar_note` поле, за яким тулза ШУКАЛА наявний запис (`georgian`/`id`), показується окремим рядком "🔎 ..." — щоб не плутати з полями, які реально передані на зміну (`add_word`/`save_grammar_note` такого рядка не мають — там усі поля належать новому запису, шукати нема що). Технічний `uuid` і поля результату, що дублюють уже показаний ввід, приховані — лишається тільки те, що справді нове (напр. повний злитий список тегів у `retag_word`). Невдалі виклики позначені ⚠️ з текстом помилки — корисно саме для діагностики випадків на кшталт обірваної по `max_tokens` відповіді, коли з тексту незрозуміло, чи дія справді відбулась
- **Теги дієслів згорнуті в один чип** — щоб панель тегів не заповнювалась окремим чипом на кожне нерегулярне дієслово, усі теги з префіксом `дієслово:` ховаються за одним агрегатним чипом "📖 Дієслова (N)". Тап показує окремий рядок під-чипів лише для конкретних дієслів (без звичайних тематичних тегів поруч) — тап на конкретне дієслово фільтрує список/повторення саме ним, повторний тап на "📖 Дієслова" згортає назад. Звичайні теги цим не зачіпаються
- **📖 Нотатки з граматики** — репетитор сам створює окрему нотатку (`save_grammar_note`: назва + повне пояснення), коли пояснює правило, яке варто зберегти для повторного звернення (особливо якщо явно попросиш "збережи це правило", або тема повторюється); перед створенням перевіряє список наявних, щоб не дублювати тему, і може підтягнути повний текст наявної нотатки (`get_grammar_note`), якщо треба процитувати. У відповіді така нотатка позначається як `[[note:ID|Назва]]` — застосунок показує це клікабельним жетоном "📖 Назва", тап відкриває окреме вікно "Нотатки" з підсвіченою потрібною карткою. Те саме посилання репетитор іноді додає і всередині тексту самої нотатки (одна нотатка згадує іншу) — рендериться так само клікабельним жетоном і в списку нотаток, і в картці повторення (`appendNoteRefs()`, спільна з чатом функція розбору `[[note:..]]`), а не голою розміткою У вікні нотаток можна їх переглядати і видаляти (🗑); текст нотатки при повторенні показується ліворуч із збереженими переносами рядків (не по центру в один рядок, як переклад слова в звичайному повторенні — та верстка не годиться для повного багаторядкового пояснення). Токенова вартість: у системний промпт додаються лише **назви** нотаток (не повний текст) — по кілька десятків токенів на нотатку, з кешуванням промпта це не змінює оцінку вартості чату
- **Виправлення нотаток через чат** — інструмент `edit_grammar_note` (пошук за `id`, той самий патч-стайл, що й у `edit_word`: передається лише поле, яке справді треба змінити) виправляє заголовок і/або зміст уже збереженої нотатки. Використовується, коли скажеш репетитору, що нотатка неповна чи містить помилку — а не повторний `save_grammar_note`, який створив би дублікат замість виправлення наявної
- **Ручне створення / редагування нотаток** — не лише через чат: у вікні "Нотатки" кнопка ➕ у шапці відкриває інлайн-форму (назва + текст), а на кожній картці кнопка ✏️ відкриває ту саму форму, заповнену наявною нотаткою. Зберігання йде на ті самі HTTP-маршрути `POST /api/notes` / `PATCH /api/notes/<id>`, які **перевикористовують** серверну логіку чат-інструментів (`execute_save_grammar_note` / `execute_edit_grammar_note` — та сама валідація, id береться з маршруту, не з тіла). Форма закривається по Escape (спершу форма, потім усе вікно) чи «Скасувати»; перехід у повторення або назад до списку теж її закриває. Після збереження список перечитується з сервера (`reloadNotes()` — БЕЗ повторного `lockBodyScroll`, інакше лічильник блокування скролу подвоївся б) і збережена нотатка підсвічується. Лише онлайн, як і решта нотаток
- **⭐ Важливі нотатки й фільтр** — на кожній картці в списку нотаток є кнопка ★/☆ (`POST /api/notes/<id>/star`, просто toggle, без чат-інструмента — позначає користувач вручну), і кнопка-фільтр в шапці вікна "Нотатки" перемикається між ☆ (показує всі) і ★ (показує лише позначені) — той самий принцип, що й на картках, тож іконка сама підказує поточний режим. Стан фільтра зберігається в `localStorage` (`notesStarredOnly`) — на відміну від розгорнутих карток 🔔 в чаті (ті скидаються при перезавантаженні), тут це стійке налаштування, щоб не перемикати щоразу заново. Фільтр не впливає на чергу повторення нотаток — лише на видимий список
- **Статистика** (кнопка "📊" в шапці) — слів усього і додано за 7 днів; картки за стадіями (нові / вивчаються (рівень 1–3) / закріплені (4+)); повторено сьогодні, серія днів поспіль 🔥, повторення за останні 7 днів. Журнал повторень по днях зберігається лише локально на пристрої (localStorage `reviewLog`, не синхронізується) — серія/графік рахуються по тому пристрою, де повторюєш
- **Офлайн-режим**: слова зберігаються також у localStorage телефону; можна відкривати аплікацію, шукати і додавати слова без з'єднання з сервером. Нові слова позначаються "⏳ не синхронізовано" і автоматично відправляються на сервер, щойно він доступний (при відкритті сторінки, поверненні мережі або по кнопці "⟳ синхр.")
- Дедуплікація при синхронізації по UUID — слово не задублюється, скільки б разів sync не повторився
- **Повторення з інтервалами (SRS)** — кнопка "🎓 Повторення" відкриває сесію карток: показується слово, згадуєш переклад, тапаєш "Показати відповідь" і оцінюєш себе ("знав"/"не знав"). Кожне слово тренується в обох напрямках (ქართული→переклад і навпаки) з незалежним прогресом. Правильна відповідь збільшує інтервал до наступного показу (1 → 3 → 7 → 14 → 30 → 60 → 120 днів), неправильна скидає на початок і повертає картку ще в цій же сесії. Нових карток максимум 10 за сесію, щоб не завалювало. Працює офлайн; прогрес синхронізується з сервером (конфлікти вирішуються за принципом "новіша оцінка виграє")
- **"🔁 Ще раз провалені"** — на екрані "Готово!" з'являється, якщо в сесії було хоч одне провалене слово (навіть якщо потім виправив його в тій же черзі): запускає новий міні-раунд лише з цих карток. Дає паузу перед повторною спробою (спершу проходиш решту сесії), замість миттєвого повтору одразу після підказки в тій самій черзі, коли відповідь ще свіжа в пам'яті і не є справжньою перевіркою. Це чиста практика: правильна відповідь у міні-раунді **не** рухає `level`/`due_at` далі (інакше дві правильні відповіді за 10 хвилин "виграли" б стільки ж днів перепочинку, скільки й справжнє повторення через реальний часовий розрив) — лише повторний провал і тут рахується по-справжньому (`level` скидається, `lapses` росте). Не натиснув кнопку — нічого не втрачено: провалене слово вже отримало `due_at` = зараз одразу при провалі, тож однаково з'явиться в наступній звичайній сесії, просто впереміш з іншими словами. Список сесійних провалів не зберігається між сесіями — це саме "добити слабкі місця зараз", а не заміна SRS-розкладу. У режимі друкування в цей список також потрапляє **слово, з яким сплутав відповідь**: якщо надрукована (неправильна для поточної картки) відповідь точно збігається з правильною відповіддю ІНШОГО слова зі словника — застосунок показує "❌ Ти написав: X — це відповідь для «Y»" і додає Y в сесійну практику (без штрафу `lapses`/`level` для Y — це лише непрямий натяк на можливу плутанину, не підтверджений провал)
- **"🎯 Тренувати: <тег>"** — кнопка під рядком повторення, з'являється лише коли вибрано тег: проходить **усю** категорію зараз (усі слова тега × обидва напрямки, `collectPractice()`), незалежно від того, чи слова взагалі due — на відміну від "🎓 Повторення", яке бере лише прострочені/нові й обмежене `NEW_PER_SESSION`. Механіка карток та сама (режими друкування/показу, "🔁 Ще раз провалені" всередині сесії працює), але сесія **нічого не пише в SRS**: ні `level`/`due_at`, ні `lapses`, ні журнал `reviewLog` (серія днів 🔥 і "повторено сьогодні" не рухаються). Причина та сама, що й для міні-раунду провалених: слово, яке ще не спливло за розкладом, згадується "по свіжій пам'яті", тож зарахувати це як справжнє повторення означало б відсунути наступний показ далі, ніж заслужено. На відміну від "🔁 Ще раз провалені", тут **і провал не карається**: туди картка потрапляє, бо була справді due і ти її справді завалив (валідний сигнал), а в тренування слова йдуть гуртом незалежно від розкладу — спроба потренувати категорію не повинна мати шансу зіпсувати реальний прогрес. Щоб не було сумнівів, у вікні видно "🎯 Тренування (без SRS)", а на фінальному екрані — "SRS не змінився". Прапорець `practiceMode` вимикається сам при старті звичайного повторення (`openSession()` — спільний старт обох сесій); яка саме відповідь іде в SRS, вирішує одна функція `writesSrs(correct)`. Тренувати "весь словник" (без тега) навмисно не можна — це дублювало б звичайне повторення, тільки без користі для розкладу
- **Позначка "проблемного слова" (leech)** — `lapses` рахує, скільки разів картку провалено за весь час (окремо для кожного напрямку — ka2uk і uk2ka незалежні), і, на відміну від `level`, ніколи не скидається успішною відповіддю. Коли лічильник у якогось із напрямків сягає порогу `LEECH_THRESHOLD` (4 — нижче за дефолт Anki (8), бо тут рахується кожен провал узагалі, включно з першим вивченням слова, а не лише регрес уже вивченого), біля слова в основному списку з'являється значок 🩹 (з підказкою при наведенні) — сигнал, що варто розібратись із цим словом окремо (напр. попросити репетитора в чаті пояснити на прикладах), а не просто далі перечитувати його на кожному повторенні. Поруч із тегами з'являється чип "🩹 проблемні (N)" — фільтрує список і сесію повторення так само, як звичайний тег, лише за критерієм leech замість тексту тега
- **Cloze-підказка з прикладу** — у режимі друкування для напрямку "переклад → ქართули" (вгадуєш грузинське слово) підказка 🧩 показує речення з поля "приклад", де саме це слово замінене на "____" — тренує слово в контексті, а не ізольовано. Працює лише коли приклад містить точний словниковий запис слова (без відмінкових/дієслівних змін форми) — якщо ні, підказка просто не з'являється
- **Повторення нотаток з граматики** — кнопка "🎓" у вікні "Нотатки" запускає окрему flip-сесію: назва нотатки → "Показати" → повний текст → сам оціни "❌ не знав"/"✅ знав". Ті самі інтервали, що й для слів (1 → 3 → 7 → ... → 120 днів), прогрес зберігається лише на сервері (нотатки й так доступні лише онлайн). Кнопка перемикається на "📖" і повертає до звичайного списку нотаток, не гублячи стан сесії
- Поле "грузинське слово" має `lang="ka"`, поле "переклад" — `lang="uk"` (правильна семантична розмітка — впливає на вимову скрін-рідерів і мову автокоректу). **Не перемикає системну клавіатуру автоматично** — перевірено на практиці: ні iOS Safari, ні мобільний Chrome не підтримують реальне перемикання розкладки клавіатури через веб-сторінку (на відміну від нативних застосунків типу Google Translate). Клавіатуру між грузинським і українським полем доводиться перемикати вручну — свідомий компроміс, транслітерацію latin→ქართული не впроваджували

## Офлайн на телефоні: разове налаштування

Service worker (механізм офлайн-кешу) браузери дозволяють лише на "безпечних" адресах — https або localhost. `http://192.168.x.x` до них не належить, тому без HTTPS аплікація працює тільки онлайн.

**HTTPS налаштовано через [mkcert](https://github.com/FiloSottile/mkcert)** — локальний центр сертифікації. Все, крім однієї дії на телефоні, автоматизовано:

- При кожному старті `app.py` сам перевіряє, чи є сертифікат для поточної IP-адреси комп'ютера, і за потреби перегенеровує його викликом mkcert (`ensure_cert()` в `server/mkcert.py`). Сертифікати лежать у `certs/` (не в git — там приватний ключ).
- Якщо на комп'ютері немає mkcert або він вимкнений — сервер тихо падає назад на `http://` (офлайн на iPhone тоді не працюватиме, але онлайн-режим лишається без змін).

**Розово на iPhone (Safari), поки є інтернет:**

1. Відкрий адресу сервера з консолі, напр. `https://192.168.100.113:5000/install-cert` — Safari завантажить профіль
2. Settings → General → VPN & Device Management → відкрий завантажений профіль → Install
3. Settings → General → About → Certificate Trust Settings → увімкни повну довіру для "mkcert ..."
4. Онови/переустанови ярлик "Add to Home Screen" на головній сторінці аплікації

Після цього кожен наступний запуск сервера з тим самим mkcert CA (навіть якщо сертифікат перегенерувався під нову IP) телефон довірятиме автоматично — крок 1–3 повторювати не треба, якщо тільки не перевстановиш mkcert CA на комп'ютері заново.

**Якщо теж користуєшся Android:** той самий трюк з `chrome://flags` більше не потрібен, але CA треба довірити і там (окремо від iPhone і від комп'ютера) — зайди на `/install-cert` у Chrome, файл `mkcert-rootCA.crt` збережеться, потім Settings → Security → Encryption & credentials → Install a certificate → CA certificate. Android покаже попередження "мережу може відстежуватись" — це очікувано для будь-якого власноруч встановленого CA, за одноразовий офлайн-доступ на своєму ж пристрої це прийнятний компроміс.

**Заходиш з іншого ПК/ноутбука в тій самій мережі (не з того, де запущений сервер):** без довіри CA браузер вважатиме сертифікат недійсним — головна сторінка ще може відкритись (після "Ваше з'єднання не приватне" → Додатково → Перейти), але окремі файли (`static/style.css`, `static/js/*.js`) підвантажуватимуться з тим самим недовіреним сертифікатом і можуть падати з помилкою на кшталт `ERR_TOO_MANY_RETRIES`. Виправлення те саме, що для Android, тільки імпорт сертифіката системний:

1. Відкрий `https://<IP>:5000/install-cert` у браузері на цьому ПК — завантажиться `mkcert-rootCA.crt`
2. `Win + R` → `certmgr.msc` → Enter
3. Зліва розгорни **Довірені кореневі центри сертифікації** → клацни на підпапку **Сертифікати**
4. Правою кнопкою на порожньому місці справа → **Усі завдання** → **Імпорт…** → обери завантажений файл → Далі → Далі → Готово (сховище вже зафіксоване, бо імпорт почато з потрібної теки — саме так, а не подвійним кліком по файлу, де можна ненароком лишити автовибір сховища, який кладе корінь CA не в довірені, і сертифікат тоді не працює)
5. Підтверди попередження системи безпеки → **Так**
6. Повністю закрий Chrome (усі вікна) і відкрий знову — інакше він ще якийсь час триматиме стару (недовірену) перевірку в кеші процесу

(Сервер віддає файл саме з розширенням `.crt`, а не `.pem`, попри те що сам mkcert зберігає корінь як `rootCA.pem` — Провідник Windows запускає майстер імпорту сертифікатів лише для `.cer`/`.crt`.)

## Архітектура синхронізації

- Кожне слово має UUID, згенерований на пристрої при створенні
- Телефон зберігає повну копію словника в localStorage і рендерить список з неї
- `POST /api/sync` приймає несинхронізовані слова, вставляє нові (`INSERT OR IGNORE` по UUID), повертає повний список — телефон замінює ним локальну копію
- Синхронізація запускається: при завантаженні сторінки, після додавання слова, при події `online`, по кнопці "⟳ синхр."
- Редагування можливе лише онлайн; видалення несинхронізованого слова — локально, синхронізованого — лише онлайн
- Service worker (`static/sw.js`): стратегія network-first — онлайн завжди свіжа сторінка з сервера (без проблеми "застряглого" кешу), офлайн — остання збережена копія. При зміні логіки кешування піднімати версію в константі `CACHE`
- Мережеві запити (в service worker і в клієнтському `fetch` для sync/delete) мають таймаут 4с (`NETWORK_TIMEOUT_MS`). Без цього при вимкненому Wi-Fi (не просто відсутньому інтернеті) `fetch()` може висіти 20-30с, поки ОС не здасться достукатись до локальної IP, перш ніж впасти в кеш/показати офлайн-статус

## Чат: налаштування (разово)

1. Зареєструйся на https://console.anthropic.com, поповни баланс (вистачить $5) і створи API-ключ
2. Скопіюй `.env.example` як `.env` (у тій же папці, що `app.py`) і встав ключ у `ANTHROPIC_API_KEY=...`
3. Перезапусти сервер

Технічні деталі:

- Модель: `claude-sonnet-5` (константа `CHAT_MODEL` в `server/chat.py`; можна замінити на `claude-opus-4-8` для глибших пояснень — дорожче)
- Системний промпт збирається з бази при кожному повідомленні: всі слова з SRS-прогресом + останні додані нетреновані до межі `CHAT_VOCAB_LIMIT` (300) + статистика; тож при великому словнику (2000+ слів) контекст не роздувається. Системний промпт кешується (prompt caching) — повторні повідомлення в межах ~5 хв платять ~10% за цю частину
- Моделі відправляються останні `CHAT_HISTORY_LIMIT` (30) повідомлень історії
- Відповідь стрімиться (`text/plain` чанки); повна відповідь зберігається в таблицю `chat_messages` після завершення. Помилки API (немає балансу, немає інтернету) показуються в чаті, але в історію не пишуться
- **Генерація відповіді йде у фоновому потоці** (`_run_chat_generation` в `server/chat.py`), незалежному від HTTP-з'єднання з браузером: якщо згорнути вкладку/додаток чи заблокувати телефон посеред відповіді, генерація однаково доводиться до кінця і зберігається в базу — при наступному відкритті чату вона вже буде на місці. Прапорець `threaded=True` в `app.run()` (app.py) обов'язковий для цього — без нього однопотоковий dev-сервер заблокував би геть усі інші запити на весь час стрімінгу. Один `_chat_lock` на застосунок гарантує, що одночасно генерується лише одна відповідь (спроба надіслати нове повідомлення, поки попереднє ще обробляється, отримує `409` з поясненням, а не плутанину в історії)
- Кнопка відправлення (`#chat-send`) показує спінер, поки триває стрімінг — видно, чи чат ще думає, чи вже закінчив
- Цикл викликів інструментів обмежений `CHAT_TOOL_LOOP_LIMIT` (5) — запобіжник від зациклення. Якщо ліміт вичерпано, поки модель ще викликала інструмент (напр. спершу перевірила наявні нотатки, тоді зберегла нову — уже 2+ кроки), користувач бачить коротке пояснення замість обірваної відповіді. Помилки виконання інструментів і будь-які неочікувані збої в чаті логуються в `chat_errors.log` (не в git) — перевір цей файл, якщо відповідь раптом обірветься без пояснення
- Ліміт `CHAT_MAX_TOKENS` (4096) — якщо модель не вклалась (напр. довге пояснення + кілька `add_word` викликів поспіль) і Anthropic повернув `stop_reason = "max_tokens"`, це **не помилка API**, тому без спеціальної обробки виглядало б як нормальне завершення — відповідь просто обривається на півслові, а обіцяні дії (наприклад, додавання слів) не встигають виконатись. Це явно детектується: у відповідь дописується попередження "обірвано — забракло токенів" і пропозиція написати "продовжуй", а сам факт логується в `chat_errors.log`
- Орієнтовна вартість на Sonnet 5: ~$0.01-0.03 за повідомлення без кешу, з кешем — менше цента; при щоденному використанні — кілька доларів на місяць
- `.env` доданий у `.gitignore` (там секретний ключ); читається з `utf-8-sig`, бо Блокнот/PowerShell пишуть BOM
- Приватність: разом із повідомленнями до Anthropic API відправляється вміст словника (слова, переклади, приклади)

## Структура

- `app.py` — тонка точка входу: створення Flask-застосунку, реєстрація модулів (`server/`), запуск сервера
- `server/db.py` — SQLite: з'єднання, схема, спільні хелпери (`get_db`, `init_db`, `normalize_tags`, ...)
- `server/words.py` — основний словник: слова, синхронізація, теги, CSV-імпорт/експорт, редагування
- `server/chat.py` — чат з репетитором (Claude): tool use, системний промпт, нотатки з граматики
- `server/mkcert.py` — генерація HTTPS-сертифіката (mkcert) для офлайн-режиму на iPhone
- `templates/index.html` — розмітка головної сторінки (форма додавання, пошук, повноекранні вікна повторення/чату/нотаток); стилі й логіка винесені окремо
- `templates/edit.html` — сторінка редагування слова (лише онлайн)
- `static/style.css` — усі стилі головної сторінки
- `static/js/` — клієнтська логіка головної сторінки, розбита по темах; підключається окремими `<script src>` в `templates/index.html` **у цьому порядку** (пізніші файли на верхньому рівні звертаються до функцій з попередніх, напр. `registerKeyboardAwareOverlay` з `overlay.js`):
  - `overlay.js` — блокування скролу фону і підгонка повноекранних вікон під `VisualViewport` (клавіатура на мобільних). `position: fixed` на `body` застосовується **лише на сенсорних пристроях** (`needsFixedBodyLock()`, перевірка `pointer: coarse`): цей хак існує проти iOS-механізму "прокрутити фокусоване поле над клавіатурою", а на десктопі скрол і так тримає `overflow: hidden`. Тримати його всюди було шкідливо: fixed-`body` стає containing block для абсолютно позиційованих елементів, які **розширення браузера вкидають у `body`** (напр. попап Google Translate при виділенні слова) — такий попап рахує координати від вьюпорта, а малюється відносно коробки `body`, а вона і вужча (`max-width: 560px`, центрована), і зсунута на `-scrollY`. Через це попап їхав убік або взагалі вилітав за верхній край екрана
  - `store.js` — localStorage, синхронізація з сервером (`sync`), дії над словами (додавання/видалення)
  - `srs.js` — повторення з інтервалами (SRS): черга карток, перевірка відповіді, оцінювання; сюди ж тренування за тегом (`startPractice`, без запису в SRS)
  - `csv.js` — експорт/імпорт CSV
  - `stats.js` — слово дня і статистика
  - `render.js` — основний рендеринг списку слів, панель тегів, перейменування тега
  - `chat.js` — чат з репетитором (Claude)
  - `notes.js` — нотатки з граматики
  - `main.js` — прив'язка обробників подій до DOM і запуск застосунку (`render(); sync();`)
- `static/sw.js` — service worker для офлайн-відкриття
- `static/manifest.json`, `static/icon-*.png` — PWA-манифест та іконки для "Add to Home Screen"
- `dictionary.db` — SQLite база, створюється автоматично при першому запуску, **не в git**
- `certs/` — HTTPS-сертифікат (`cert.pem`, `key.pem`) та відмітка IP, для якого він згенерований (`meta.txt`); генерується автоматично, **не в git**
- `.env` — секрети (`ANTHROPIC_API_KEY`), **не в git**; зразок — `.env.example`
- `chat_errors.log` — помилки чату з репетитором (збій API, виконання інструментів, обрив стріму), створюється автоматично, **не в git**
- `start.bat` — запуск подвійним кліком на Windows
- `backup_db.py`, `backup_db.bat` — бекап `dictionary.db` у `backups/` (не в git)
- `restore_db.py`, `restore_db.bat` — відновлення `dictionary.db` з бекапу (найновішого або вказаного); стару базу архівує, не видаляє

## HTTPS / mkcert

Потрібен, щоб service worker (а отже й офлайн-режим) працював на iPhone — Safari дозволяє SW лише в "безпечному контексті" (https), без винятків для локальної мережі на відміну від десктопного Chrome.

- Встановлено через `winget install FiloSottile.mkcert`, довірений локальний CA створений командою `mkcert -install` (одноразово на цьому комп'ютері)
- `app.py` при кожному старті сам генерує/перегенеровує лист-сертифікат для поточної IP-адреси через `ensure_cert()` — ручного втручання не треба, навіть коли роутер видає новий IP
- `/install-cert` віддає лише публічний корінь CA (не приватний ключ) — безпечно віддавати будь-якому пристрою у своїй мережі, з ним не можна підписувати нові сертифікати. Файл на диску в mkcert називається `rootCA.pem`, але сервер віддає його під іменем `mkcert-rootCA.crt` (`download_name` в `server/mkcert.py`) — Windows асоціює майстер імпорту сертифікатів лише з `.cer`/`.crt`, а не `.pem`
- Якщо mkcert не знайдено на машині — сервер автоматично падає назад на звичайний `http://` (без крашу), офлайн на iPhone просто не працюватиме

## База даних

Таблиця `words`: `id, uuid, georgian, translation, example, tags, created_at`. Таблиця `reviews` (SRS-прогрес): `word_uuid, direction ('ka2uk'|'uk2ka'), level, due_at, reviewed_at, lapses`, первинний ключ `(word_uuid, direction)`; видалення слова каскадно чистить його оцінки. `lapses` — лічильник провалів за весь час (leech-позначка), на відміну від `level` ніколи не скидається. Таблиця `chat_messages` (історія чату): `id, role ('user'|'assistant'), content, created_at`. Таблиця `grammar_notes`: `id, title, content, created_at`. Таблиця `note_reviews` (SRS-прогрес повторення нотаток, ті самі інтервали, що й слова): `note_id (PK), level, due_at, reviewed_at` — лише на сервері, без офлайн-копії; видалення нотатки каскадно чистить і її прогрес. Схема створюється автоматично, прості міграції — в `init_db()` в `server/db.py` (там же бекфіл uuid для старих записів).

## Ідея на майбутнє: Docker для розгортання іншими людьми

Застосунок за задумом однокористувацький (одна SQLite-база, один `.env` з персональним `ANTHROPIC_API_KEY`, mkcert-сертифікат під конкретну локальну IP) — тому для когось іншого, хто захоче собі таку ж аплікуху, природний варіант "кожному своя приватна копія", не спільний сервер (останнє вимагало б автентифікації й ізоляції даних між користувачами — це вже інший проєкт).

Docker знизив би поріг входу для нетехнічної людини (замість "постав Python, створи venv, встанови залежності" — один `docker compose up`). Головна пастка, яку варто врахувати одразу, а не виявити постфактум: SQLite-файл і `.env` мають лежати на **volume/bind mount**, змонтованому ЗЗОВНІ образу — інакше оновлення застосунку (перебілд контейнера) стирає весь прогрес користувача разом зі старим образом. Це розв'язна проблема (саме для цього й існують volumes — так само працює офіційний Postgres/MySQL образ), але це додатковий крок налаштування, який легко забути чи зробити неправильно, тож requires чіткої інструкції/дефолтного `docker-compose.yml`, а не просто "ось Dockerfile".

Поки що свідомо не займаємось — жодного реального запиту на це ще не було, і додається складність заради гіпотетичного майбутнього користувача.
